INCIDENCIAS INSTALACION WINDOWS SERVER 2019 - ATLAS BALANCE
Version objetivo: V-01.04
Fecha: 2026-04-25

Objetivo
-------
Registrar los errores, bugs, cambios operativos e incidencias detectadas durante la instalacion real en Windows Server 2019 para que se corrijan o se tengan en cuenta antes de publicar la siguiente version.

Estado de cierre en fuente
--------------------------
Corregido en V-01.04:
- Validacion de carpeta instalable vs carpeta fuente.
- Documentacion de ZIP `main` no instalable.
- PostgreSQL 17 documentado como valido.
- Fallback claro cuando `winget` falla.
- Wrapper `install.cmd`/`scripts/install.ps1` revisado.
- PFX regenerado en reinstalacion.
- Health check con `curl.exe -k`.
- Certificado cliente y lectura de credenciales iniciales documentados.
- Deteccion de BD existente para no escribir password admin falsa.
- Script soportado `scripts\Reset-AdminPassword.ps1`.
- Cabeceras `SAMEORIGIN` y `frame-ancestors 'self'` mantenidas.
- `Build-Release.ps1` incluye scripts operativos nuevos.

Pendiente fuera de esta maquina:
- Probar `AtlasBalance-V-01.04-win-x64.zip` en Windows Server 2019 real.
- Publicar el ZIP como asset de GitHub Release cuando la validacion real termine.
- Rotar cualquier secreto que haya sido expuesto durante soporte.

Nota de seguridad
-----------------
No documentar passwords, tokens ni secretos reales. Durante la instalacion se compartio una password de PostgreSQL por chat; debe rotarse en el servidor. Este documento NO incluye ese valor.


1. Confusion entre carpeta fuente y paquete instalable
-----------------------------------------------------
Sintoma:
- Se intento instalar desde `AtlasBalance-main`, una carpeta con la estructura completa del repositorio.
- El instalador solo funciona correctamente desde el paquete release que contiene `api/GestionCaja.API.exe`, `watchdog/GestionCaja.Watchdog.exe`, `scripts` y wrappers `.cmd`.

Causa:
- La documentacion y el flujo operativo no distinguen con suficiente dureza entre repositorio fuente y paquete instalable.

Solucion para V-01.04:
- El instalador debe validar al inicio que esta dentro de un paquete release.
- Si no existen `api/GestionCaja.API.exe` y `watchdog/GestionCaja.Watchdog.exe`, debe parar con un mensaje claro:
  "Esta carpeta no es el paquete instalable. Genera o descarga AtlasBalance-V-XX-win-x64.zip."
- La documentacion de usuario debe decir explicitamente que el ZIP `main` de GitHub no sirve como instalador de servidor.

Verificacion esperada:
- Ejecutar instalador desde carpeta fuente debe fallar con mensaje claro.
- Ejecutar instalador desde `AtlasBalance-V-01.04-win-x64` debe continuar.


2. Paquete disponible V-01.03 mientras la version actual es V-01.04
------------------------------------------------------------------
Sintoma:
- En disco estaba generado/publicado `AtlasBalance-V-01.03-win-x64`.
- La version activa del proyecto era `V-01.04`.

Causa:
- La version de codigo y el paquete instalado no estaban alineados.

Solucion para V-01.04:
- Regenerar paquete `AtlasBalance-V-01.04-win-x64.zip`.
- Publicarlo como asset de GitHub Release.
- No instalar V-01.03 si el objetivo es validar V-01.04.

Verificacion esperada:
- `C:\AtlasBalance\VERSION` debe mostrar `V-01.04`.
- La pantalla/configuracion de version debe coincidir con `Atlas Balance/VERSION`.


3. PostgreSQL 17 en Windows Server 2019
--------------------------------------
Sintoma:
- Se instalo PostgreSQL 17 y se pregunto si era compatible.

Causa:
- La documentacion hablaba de PostgreSQL 16+ pero el flujo estaba mentalmente centrado en 16.

Solucion para V-01.04:
- Documentar que PostgreSQL 17 es valido porque el requisito real es PostgreSQL 16+.
- Mantener `PostgresBinPath` autodetectando 17 antes que 16.
- En Windows Server 2019, recomendar instalacion manual de PostgreSQL en vez de depender de `winget`.

Verificacion esperada:
- `psql.exe -h localhost -p 5432 -U postgres -d postgres -c "SELECT version();"` devuelve PostgreSQL 17.
- Instalador acepta `-PostgresBinPath "C:\Program Files\PostgreSQL\17\bin"`.


4. `winget` no debe ser el camino principal en Windows Server 2019
-----------------------------------------------------------------
Sintoma:
- El instalador puede intentar preparar PostgreSQL con `winget`.

Causa:
- En Windows Server 2019, `winget` y dependencias de App Installer/Store no son una base fiable para produccion.

Solucion para V-01.04:
- Documentacion: recomendar PostgreSQL manual en Windows Server 2019.
- Instalador: si falla `winget`, mostrar instrucciones concretas para instalar PostgreSQL manualmente.
- No vender "one click" como garantia en Server 2019 limpio.

Verificacion esperada:
- Instalacion manual de PostgreSQL + `Instalar-AtlasBalance.ps1` funciona sin `winget`.


5. Wrapper `install.cmd` / `scripts/install.ps1` paso mal argumentos
--------------------------------------------------------------------
Sintoma:
- Error:
  "No se puede convertir el valor C:\AtlasBalance al tipo System.Int32" para `ApiPort`.

Causa probable:
- El wrapper `install.cmd`/`scripts/install.ps1` de V-01.03 reenvio argumentos de forma fragil desde PowerShell/CMD.

Solucion para V-01.04:
- Revisar y simplificar el wrapper.
- Opcion preferida: `install.cmd` debe llamar al instalador real con argumentos preservados de forma no ambigua.
- Documentar alternativa soportada:
  `powershell.exe -NoProfile -ExecutionPolicy Bypass -File ".\scripts\Instalar-AtlasBalance.ps1" ...`

Mitigacion usada:
- Saltarse `install.cmd` y ejecutar directamente:
  `scripts\Instalar-AtlasBalance.ps1`

Verificacion esperada:
- `.\install.cmd -InstallPath C:\AtlasBalance -ServerName SRVGESTIO2019 -ApiPort 443 -DbPort 5432 -PostgresAdminPassword <password> -PostgresBinPath "C:\Program Files\PostgreSQL\17\bin"` no desordena parametros.


6. Reinstalacion rompia certificado HTTPS PFX
---------------------------------------------
Sintoma:
- `AtlasBalance.API` quedaba `Stopped`.
- Windows Event Log:
  `System.Security.Cryptography.CryptographicException: La contraseña de red especificada no es válida`
  al cargar `atlas-balance.pfx`.

Causa:
- `Instalar-AtlasBalance.ps1` reutilizaba `C:\AtlasBalance\certs\atlas-balance.pfx` si ya existia.
- En cada reinstalacion generaba una password HTTPS nueva y la escribia en `appsettings.Production.json`.
- Resultado: PFX viejo + password nueva.

Solucion aplicada en fuente V-01.04:
- `New-AtlasCertificate` elimina `atlas-balance.pfx` y `atlas-balance.cer` existentes antes de generar certificados nuevos.

Mitigacion usada en servidor:
- Parar servicios.
- Borrar:
  `C:\AtlasBalance\certs\atlas-balance.pfx`
  `C:\AtlasBalance\certs\atlas-balance.cer`
- Relanzar `Instalar-AtlasBalance.ps1`.

Verificacion esperada:
- `Get-Service AtlasBalance.API,AtlasBalance.Watchdog` muestra ambos `Running`.
- `curl.exe -k https://SRVGESTIO2019/api/health` devuelve `HTTP/1.1 200 OK`.


7. `Invoke-WebRequest` dio falso negativo en health check HTTPS
--------------------------------------------------------------
Sintoma:
- `Invoke-WebRequest` / `Invoke-RestMethod` devolvian:
  "Se ha terminado la conexion: Error inesperado de envio."
- Pero `curl.exe -k -v https://SRVGESTIO2019/api/health` devolvia `HTTP/1.1 200 OK`.

Causa:
- PowerShell/WinHTTP antiguo en Windows Server 2019 puede fallar con TLS/certificado autofirmado aunque Kestrel este bien.

Solucion para V-01.04:
- En documentacion de instalacion, usar `curl.exe -k` como prueba primaria de health check.
- No concluir que la API esta caida solo por `Invoke-WebRequest`.

Verificacion esperada:
- `curl.exe -k -v https://localhost/api/health`
- `curl.exe -k -v https://SRVGESTIO2019/api/health`


8. Certificado cliente no confiado
----------------------------------
Sintoma:
- Navegador muestra advertencias o errores de red en operaciones frontend.

Causa:
- Certificado self-signed generado por Atlas no estaba instalado como raiz confiable en el servidor/cliente.

Solucion operativa:
- Instalar:
  `C:\AtlasBalance\certs\atlas-balance.cer`
  en `Entidades de certificacion raiz de confianza`.

Comando:
- En servidor o cliente, como administrador:
  `certutil -addstore -f Root "C:\Ruta\atlas-balance.cer"`

Solucion para V-01.04:
- Documentar paso obligatorio para clientes.
- Incluir diagnostico: si `/api/health` responde con `curl.exe -k` pero navegador falla, revisar confianza del certificado.


9. Archivo de credenciales iniciales protegido
----------------------------------------------
Sintoma:
- Al abrir `C:\AtlasBalance\INSTALL_CREDENTIALS_ONCE.txt`, Windows decia que no habia permisos.

Causa:
- El instalador restringe ACL a Administradores/SYSTEM. Correcto por seguridad, pero poco explicado.

Solucion para V-01.04:
- Documentar que debe abrirse en PowerShell elevado:
  `Get-Content "C:\AtlasBalance\INSTALL_CREDENTIALS_ONCE.txt"`
- Si hiciera falta:
  `icacls "C:\AtlasBalance\INSTALL_CREDENTIALS_ONCE.txt" /grant *S-1-5-32-544:F`

Verificacion esperada:
- Administrador local puede leer el archivo.
- Tras primer login y cambio de password, el archivo se borra.


10. Credenciales iniciales invalidas tras reinstalar sobre una BD existente
--------------------------------------------------------------------------
Sintoma:
- Login muestra "Credenciales invalidas" usando `INSTALL_CREDENTIALS_ONCE.txt`.

Causa:
- `SeedData` solo crea el admin inicial si `USUARIOS` esta vacia.
- En reinstalacion sobre una base existente, el instalador regenera `INSTALL_CREDENTIALS_ONCE.txt`, pero el admin real ya existe con su password anterior.
- El archivo nuevo puede mentir. Eso es mal UX y mala operacion.

Solucion para V-01.04:
- Si la BD ya tiene usuarios, el instalador NO debe escribir "Password admin inicial" nueva como si fuera valida.
- Debe detectar usuarios existentes y mostrar:
  "Base existente detectada. Las credenciales iniciales no se regeneran. Usa el admin ya creado o ejecuta un reset de password."
- Agregar un script soportado:
  `scripts\Reset-AdminPassword.ps1`
  que resetee una cuenta admin de forma controlada.

Mitigacion usada:
- Reset manual en PostgreSQL con `pgcrypto`:
  - Actualizar `password_hash`
  - `primer_login = true`
  - `activo = true`
  - `failed_login_attempts = 0`
  - `locked_until = NULL`
  - rotar `security_stamp`
  - revocar refresh tokens

Verificacion esperada:
- Login con password temporal funciona una sola vez.
- App obliga a cambiar password.
- Tokens anteriores quedan revocados.


11. Error SQL por comillas al actualizar tabla `USUARIOS`
---------------------------------------------------------
Sintoma:
- PostgreSQL devolvio:
  `ERROR: no existe la relacion usuarios`

Causa:
- Las tablas del proyecto estan en mayusculas con comillas: `"USUARIOS"`.
- Al pasar SQL con `-c` desde PowerShell, las comillas se perdieron o no llegaron como se esperaba.

Solucion operativa:
- Enviar SQL por stdin con `psql -f -`:
  `$sql | psql.exe ... -f -`

Solucion para V-01.04:
- No pedir al operador que escriba SQL largo a mano.
- Crear script `Reset-AdminPassword.ps1` con parametros seguros.


12. Bloqueo por demasiados intentos de login
--------------------------------------------
Sintoma:
- Mensaje:
  "Demasiados intentos. Espera unos minutos."

Causa:
- Se probaron credenciales incorrectas varias veces.
- El rate limit/bloqueo de login hizo su trabajo.

Solucion operativa:
- Limpiar:
  `failed_login_attempts = 0`
  `locked_until = NULL`
  para el usuario afectado.

Solucion para V-01.04:
- El script `Reset-AdminPassword.ps1` debe limpiar bloqueo y contador.
- Documentar que el bloqueo operativo dura 30 minutos si no se limpia manualmente.

Verificacion esperada:
- Tras reset, `failed_login_attempts = 0` y `locked_until IS NULL`.


13. Modal `Importar movimientos` rechazado por cabeceras anti-frame
-------------------------------------------------------------------
Sintoma:
- La app carga y permite login.
- En dashboard de cuenta, al pulsar `Importar movimientos`, aparece un modal gris con icono de documento roto/rechazo de conexion.

Causa:
- El frontend abre `/importacion` dentro de un `iframe`.
- La API envia:
  `X-Frame-Options: DENY`
  `Content-Security-Policy: frame-ancestors 'none'`
- La app se bloquea a si misma. Esto no es seguridad, es pisarse el cable.

Solucion aplicada en fuente V-01.04:
- `X-Frame-Options: SAMEORIGIN`
- `frame-ancestors 'self'`
- Se mantienen bloqueados embebidos externos.

Mitigacion usada en V-01.03 instalado:
- Parchear el bundle servido para que el boton navegue a `/importacion` como pagina completa, evitando el iframe.

Pendiente:
- Regenerar/publicar V-01.04 para restaurar el modal embebido sin parchear JS minificado.

Verificacion esperada:
- Pulsar `Importar movimientos` abre el modal correctamente.
- Las cabeceras permiten frame same-origin pero no cross-origin.


14. Parche manual del bundle JS no debe quedar como solucion permanente
----------------------------------------------------------------------
Sintoma:
- Tras parchear `index-*.js`, importacion funciona pero abre como pagina completa, no como ventana/modal dentro de la app.

Causa:
- El parche operativo sustituyo `b(!0)` por navegacion a la ruta de importacion para saltarse el iframe bloqueado.

Solucion para V-01.04:
- No mantener este parche como arquitectura.
- Publicar backend con cabeceras corregidas.
- Restaurar comportamiento modal original desde fuente.

Verificacion esperada:
- No existen `.bak-iframe-fix` como artefactos activos.
- El bundle se genera desde `npm run build`, no con edicion manual.


15. Password de PostgreSQL expuesta durante soporte
---------------------------------------------------
Sintoma:
- El operador escribio la password real de PostgreSQL en el chat.

Causa:
- Se pidio una ejecucion rapida y se intento dejar comandos "listos".

Solucion operativa:
- Rotar password del usuario `postgres`.
- Revisar si hay scripts, historial de PowerShell o capturas que la contengan.

Solucion para V-01.04:
- Nunca pedir que se pegue una password real en el chat.
- Usar `Read-Host -AsSecureString` cuando sea posible.
- Si se necesita automatizar, usar variables de entorno temporales o prompt interactivo.
